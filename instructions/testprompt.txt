Nios UART
UART

I've used this initialism (maybe it's a acronym), but a UART (pronounced you-art) is a Universal Asynchronous Receiver/Transmitter. Very, very occasionally, you will hear about a USART (where the 'S' stands for "Synchronous").  As I've mentioned before, it's an old, simple, but still very popular industrial standard.

It's a very simple protocol. It's Full Duplex, asynchronous, and generally a peer-to-peer protocol. You may find it weird to call it a peer-to-peer protocol when it often hooks a keyboard up to the microcontroller. Really, you, as the user behind the keyboard, are connected as the peer.

Anyways, this is a serial protocol that uses as many as four pins.

"Wait!" you say. "One bit at a time over four pins? How that can be serial?"

Two of the pins are for data and there is just one bit going in each direction, so it's still serial. One of them is the RXD (Receive Data), which is the input pin for the Receive Shift Register. The other is the TXD (Transmit Data), which is the output pin for Transmit Shift Register.

The other two pins are for something called "flow control". Flow control is a really cool ability that we provide to the receiver that allows it to tell a transmitter to just sit down and shut up for a while.  

I can't even

It makes the sender ask for permission before sending data. It does this by first asserting the RTS (Request to Send) signal when it wants to talk to the receiver. On the other end, the potential receiver is listening for that. If it's ready to handle some data, it asserts the CTS (Clear to Send) signal. If it just can't even right now, it won't assert that signal, leaving the sender to wait until conditions are more favorable for the receiver.

It's relatively unusual to find an RS-232 serial link that actually uses flow control. The bit rates just aren't that high in practice. But most UARTs still support that portion of the standard.

The Registers for the Nios UART

This is the full register file. All the registers are 16 bits. Note that there are separate registers for the Receive data and the Transmit data. You'll either interact with those registers as 8-bit pointers or use 16-bit pointers and mask off that upper byte.

Let's look at all those flags in the Status register.

Some UART Status flags

Look! Flags for a bunch of the errors we talked about: framing errors, break errors, parity errors, and overrun errors.

More UART Status register flags

Here are the flags that we talked about before. Instead of "RDRF", this one is called RRDY. The "TDRE" flag is called TRDY. There's one more flag that's a bit different -- the TMT flag. This one tells you that not only is the Transmit Data register empty, but so is the Shift register. In short, the target device should have all of the data you tried to send.

Final Status flags

There's a flag that tells you that you received the CTS signal from the target device. Perhaps the more interesting flag is the EOP. If a higher-level protocol is running on top of the UART. It will be sending bigger "packets" as a collection of bytes. We can set a value that indicates that we received a special character designated to end that packet and terminate the communication. The EOP flag says we received that character. The endofpacket register lets you specify what that character is.

UART Control Register flags

The TBRK flag is in the Control register. It makes the UART send a Break character.

More UART Control flags

This flag makes the UART assert the RTS signal if you're using flow control.

The last register of note is the divisor register. Here's where you set the bit rate (if you didn't set it in Qsys). The bit rate is divided down from Avalon clock by whatever value you place into the register. (Well, almost.) The exact equation is:
Bit Rate = Avalon clock frequency / (divisor + 1)

Provided you didn't set the bit rate in Qsys, the software can alter the bit rate at run time. This isn't a weird or useless feature; it allows one UART to support different client devices .

So just as an exercise, consider a simple terminal echo program. The user types a character that gets transmitted to the UART, where it gets passed up to the main program to do something (ostensibly). But it also echos the character back to the original transmitter so it can be displayed on the local screen. Here's what the code might look like:

#include <alt_types.h>
#define bitrate 28800
#define sysclock 50000000
int main()
{
  // I didn't double check, but I think these registers
  // are word-aligned as well.
  alt_u32* uartBase = 0x10004000; // or whatever Qsys assigns
  alt_u8 charData;
  alt_u16 portStatus;

  // Maybe other initializations besides this
  *(uartBase + 4) = sysclock / bitrate - 1;

  while (1)
  {
    // Wait for a character
    portStatus = *(uartBase + 2) & 0x80;
    while (0 == portStatus)
    {
      portStatus = *(uartBase + 2) & 0x80;
    }
    // If we're here, RRDY is set
    charData = (alt_u8) *uartBase & 0xFF;
    // Don't need to explicitly clear the flag because
    // reading from rxdata did that for us automagically
    // Hopefully there's something else to do with the char.
    // Now poll to echo the character
    portStatus = *(uartBase + 2) & 0x40;
    while (0 == portStatus)
    {
      portStatus = *(uartBase + 2) & 0x40;
    }
    *(uartBase + 1) = charData;
  }
}
